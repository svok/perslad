# Сквозные тесты для управления контекстом

## Обзор

Набор сквозных тестов для проверки динамической компрессии контекста в LangGraph Agent. Эти тесты покрывают различные сценарии переполнения контекста и обеспечивают корректность работы системы.

## Тесты

### 1. `test_compress_context_when_near_limit`

**Цель**: Проверить динамическую компрессию контекста при приближении к лимиту.

**Сценарий**:
1. Создается длинная история сообщений (31 пара: 15 сообщений пользователя + 15 сообщений ассистента)
2. Контекст приближается к лимиту 16384 токенов
3. Агент применяет динамическую компрессию
4. Запрос должен пройти успешно

**Проверяемый функционал**:
- Корректное определение момента приближения к лимиту (margin_of_error = 500 токенов)
- Применение SummarizedContext стратегии
- Агрессивное уменьшение RAG контекста до 1024 токенов
- Успешный ответ от LLM после сжатия

**Статус**: ✅ PASSED

---

### 2. `test_context_overflow_with_rag_context`

**Цель**: Проверить обработку переполнения при наличии RAG контекста.

**Сценарий**:
1. Создается длинная история (41 сообщение)
2. RAG контекст добавляется в system prompt
3. Контекст превышает лимит
4. Агент должен либо сжать контекст, либо откатиться к fallback

**Проверяемый функционал**:
- Подсчет RAG токенов с учетом контекста
- Обработка переполнения
- Отсутствие падений с vllm.exceptions.VLLMValidationError

**Статус**: ✅ PASSED

---

### 3. `test_mixed_scenario_normal_and_overflow`

**Цель**: Проверить смешанный сценарий с нормальным диалогом и переполнением.

**Сценарий**:
1. Нормальный диалог (9 сообщений)
2. Длинная история (50 сообщений)
3. Финальный запрос

**Проверяемый функционал**:
- Разделение на нормальные и проблемные участки
- Применение компрессии там, где необходимо
- Поддержание работоспособности после сжатия

**Статус**: ✅ PASSED

### 4. `test_incremental_context_growth`

**Сценарий**:

**Проверяемый функционал**:

- Устойчивость системы после изменения размера контекста

**Статус**: ✅ PASSED

### 6. `test_multiple_context_overflow_recovery`

**Цель**: Проверить восстановление после нескольких случаев переполнения.

**Сценарий**:

**Статус**: ✅ PASSED

## Анализ работы системы

### Детекция переполнения:
```python
margin_of_error = 500  # Если total > limit - 500, применяем компрессию
```

### Стратегии компрессии:
1. **Normal**: Использует ContextBuilder с max_context_tokens=2048
2. **Compressed**: Использует ContextBuilder с max_context_tokens=1024
3. **Overflow**: Полный откат RAG контекста

### Логирование:
```
⚠️ Context close to limit: 15800 tokens, limit=16184, applying compression...
✂️ Context compressed: 16000 -> 12500 tokens
```

## Частые проблемы и решения

### Проблема: Тесты становятся нестабильными при задержках
**Решение**: Увеличить timeout до 180 секунд, добавить retry логику

### Проблема: Компрессия слишком агрессивная
**Решение**: Увеличить margin_of_error до 1000 токенов

### Проблема: LLM ответы слишком короткие после сжатия
**Решение**: Проверить сохранение контекста в system prompt
