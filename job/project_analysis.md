## 1. Текущая Архитектура (As-Is)

Проект Perslad представляет собой модульную систему на базе микросервисов, развернутую в Docker.

*   **Core Stack:** Python 3.12+, Docker Compose.
*   **LLM Serving:** `vLLM` (локальный инференс, OpenAI API compatible). Отличный выбор, соответствует стандартам рынка.
*   **Orchestration:** `LangGraph` (`StateGraph`). Реализован базовый паттерн ReAct (Agent -> Tools -> Agent).
*   **RAG & Data Layer:**
    *   **Vector:** `pgvector` (PostgreSQL) — активное хранилище эмбеддингов.
    *   **Graph:** **NebulaGraph** (v3.8.0). Инфраструктура полностью описана (`docker-compose-nebula-lite.yml`) и протестирована (`test_nebula.py`), но **пока не подключена к пайплайну индексации**.
*   **Tools:** Реализация MCP-серверов (`mcp-bash`, `mcp-project`), но интеграция выглядит проприетарной.

## 2. Сравнение с рынком (Gap Analysis)

| Область | Текущее состояние (Perslad) | Требование рынка (2026) | Статус (Gap) |
| :--- | :--- | :--- | :--- |
| **RAG** | **Hybrid-Ready (Dormant).** Инфраструктура для GraphRAG (Nebula) есть, но данные льются только в вектор. | **Active GraphRAG.** Граф знаний должен строиться автоматически при индексации кода (AST, вызовы). | 🟡 **Integration Gap** (Технически все готово, нужно только связать) |
| **Agents** | **Single Loop.** Один агент в цикле (ReAct). Линейное выполнение. | **Multi-Agent / Hierarchical.** Специализированные агенты (Planner, Coder) работающие параллельно. | 🟠 **Moderate Gap** |

| **Protocol** | **Custom MCP.** Реализация серверов есть, но неясно, используется ли стандартный клиент для подключения внешних MCP. | **Native MCP.** Возможность подключить *любой* публичный MCP-сервер (Github, Slack, Filesystem) одной строкой конфига. | 🟡 **Minor Gap** |

## 3. Сильные стороны (Assets)
1.  **NebulaGraph:** Наличие такой мощной распределенной ГСУБД "из коробки" — огромное преимущество перед проектами на Neo4j/NetworkX. Это позволяет масштабироваться на миллионы узлов (огромные репозитории).
2.  **Инфраструктура:** Грамотно настроенный `vLLM` и Docker-среда.
3.  **LangGraph:** Использование правильного фреймворка (LangGraph вместо устаревших LangChain Chains).

## 4. Ключевые проблемы (Liabilities)
1.  **Disconnection:** Мощный движок NebulaGraph простаивает. Агент "слеп" к структуре кода, хотя база данных для прозрения уже стоит рядом.
2.  **Lack of Metrics:** Отсутствие измеримого качества ответов.
3.  **Простой граф:** Агент пытается делать всё сам (и думать, и искать, и писать код), что противоречит тренду "Think Less, Search More" (специализация).

## 5. Рекомендации (Preview of Strategy)
1.  **Activate GraphRAG:** Главный приоритет — написать адаптер для `ingestor`, который будет парсить код (через `tree-sitter` или LLM) и сохранять узлы/связи в NebulaGraph.
2.  **Refactor Agent:** Научить агента делать Cypher(nGQL)-запросы в Nebula для сложных вопросов ("кто вызывает функцию X?").
3.  **Evals First:** Перед усложнением агентов настроить систему оценки (Evals), чтобы измерять прогресс.
